# 记忆系统三层自主架构

记忆系统应该**自己管好自己**，不依赖外部 cron 做 janitor。

## 三层架构概览

```
Layer 3: 自主思考（每周日 18:00，agent-mode）
   记忆反思 → 发现模式/矛盾/缺口 → 写入反思报告
   ↑ 真正的 LLM 推理，不是关键词匹配

Layer 2: 自动进化（每日 06:00，no_agent）
   收集记忆+信念 → 结构化快照 → 触发 Reflect → 写入蒸馏日报
   ↑ 为 Layer 3 准备数据原料

Layer 1: 自我管理（每次写入，内嵌规则）
   容量自检 >80%→合并 | instinct 同域≥3→升格skill | user 去重替换
   ↑ 不依赖任何外部 cron，写入时自洽
```

## Layer 1: 自我管理

每次向 `memory` 或 `user` 写入时执行的内嵌规则：

### 容量自检
- 写入前检查目标容量。>80% 时先合并同域条目再写入
- 合并策略：保留置信度最高的版本，删除其余同域项
- 如果同域无合并候选 → 跳过合并直接写入（不阻塞写入）

### Instinct 自动毕业
- 写入新 instinct 前检查：同 domain 是否已有 ≥3 条？
- 是 → 自动创建 skill（`skill_manage(action="create")`），从 memory 删除所有原始 instinct 条目
- 单条 confidence ≥0.85 也触发自动毕业
- 参考 `continuous-learning` skill 的阶段 6 获得完整规则

### User 画像去重
- 写入 user profile 前搜索是否已有同类条目
- 有 → 用 `memory(action="replace")` 替换而非追加
- 同类判定：核心关键词重叠率 >50%

### 不依赖外部 cron
- 清理是每次写入的一部分，不是独立任务
- 不要为"memory 满了"设告警——它不应该满
- 维护 cron 做的不是清理是学习（Layer 2 + Layer 3）

## Layer 2: 自动进化（每日 06:00）

`memory-tmt-cluster.py` — no_agent 模式，静默运行

**工作流程：**
1. 读取 Amber 所有记忆（`GET /memories`）
2. 按主题粗分类（审计/覆盖率/版权/Docker/错误模式）
3. 生成结构化快照 → 写入 `60_蒸馏日报/YYYY-MM-DD-记忆快照.md`
4. 触发 Reflect（`POST /reflect?mode=light`）→ 热度衰减 + 层级迁移
5. 输出 `[SNAPSHOT]` 或 `[SILENT]`

## Layer 3: 自主思考（每周日 18:00）

agent-mode cron，使用 LLM 推理分析 L4 预处理数据 + Layer 2 快照

**数据输入：**
- TMT L4 结构化周摘要（周日 17:00 预处理）
- Layer 2 本周快照
- 会话归档

**三次推理：**
1. **模式发现** — 本周重复主题、跨会话趋势、值得关注的变化
2. **矛盾检测** — 信念之间的冲突、信念与记忆的不匹配、置信度异常
3. **缺口识别** — 高频主题无信念覆盖、用户常问但无答案的知识

**输出：** `60_蒸馏日报/YYYY-MM-DD-反思报告.md`

**关键原则：**
- 不是逐条摘要——要提取跨记忆的洞察
- 矛盾检测需报告真正的冲突，非语言细微差异
- 无发现时输出 `[SILENT]`

## Layer 2.5：进化回写层（日/周报的反馈环路）

Layer 2（数据处理）和 Layer 3（语义推理）之间的进化闭环。让日/周/月报不只是输出工具，而是**进化验证器**。

```
L4（TMT 结构化周摘要）──→ Layer 3（LLM 语义推理）
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
                ① 写报告     ② 校正      ③ 标记缺口
                              instinct
                    │           │           │
                    ▼           ▼           ▼
                周报（周五）──→ 对比上周预测 → memory 回写
                                        │
                                        ▼
                              instinct 置信度更新
```

**3 条核心回写环路：**

| 回写 | 触发 | 动作 | 影响 |
|------|------|------|------|
| **预测验证** | 写周报时 | 对比上周 Layer 3 预测与实际进展 → 对应 instinct 置信度调升/调降 | instinct 从 LLM 推理获得校正信号 |
| **模式信号** | 周报发现异常趋势 | `memory(action="add")` 新 instinct 或更新 | 异常在周期性回顾中被发现 |
| **知识老化** | 每次回写时 | 检查 >21 天未引用条目 → 标记到 `70_待确认/知识老化.md` | 记忆保留决策动态化 |

**写周报/月报前必须执行的检查清单：**
```python
# 1. 上周反思报告存在？→ 对比预测 → memory 更新置信度
# 2. 本周有重复异常模式？→ 记录 trend instinct
# 3. 有超过 21 天未引用条目？→ 标记老化
```

## Layer 0：功能使用率监控（地基层）

所有三层之上持续运行的自我审计层。**没有使用率数据，设计缺陷永远不被发现。**

**做法：** 每个 API 端点记录调用次数，每周自动生成使用率报告。

### 每周输出
- 调用次数 = 0 的端点 → 标记为可能死代码
- 调用次数 < 5 且非健康检查 → 标记为低价值
- 连续 3 周零调用 → 写入 memory 作为弃用候选

### 原则
组件的存在价值必须被持续验证，而非设计时假设。功能驱动而非需求驱动是过度设计的根因。

## 双通道实时同步（精简版）

对话中发现画像级事实时立即触发，不走 cron：

```bash
python ~/AppData/Local/hermes/scripts/memory-dual-sync.py realtime '{"fact":"事实内容","confidence":0.95}'
```

**双通道：**
1. **Honcho Peer Card** — 追加/替换画像事实（自动去重+合并同类）
2. **Vault 画像日志** — `10_事实/YYYY-MM-DD-画像日志.md`

**为什么不是三通道：** 第三通道（Amber Belief）从未产生用户可感知的价值。N 组件需要 N×(N-1)/2 条同步路径——每增一个通道，故障面指数增长。双通道足够。

**触发规则：** `identity` / `tool_stack` / `preference` / `project_decision` / `quality_standard` / `knowledge_insight`

## 三个自主检查框架

每次涉及记忆系统的改进方案，必须以"三个自主"为北极星审计每个组件：

| 自主 | 审计问题 | 不通过时的裁减方向 |
|------|---------|-----------------|
| **自主管理** | 这个功能是否能**自己管理自己的生命周期**？是否需要用户介入清理/维护？ | 如果依赖用户维护，要么加自检规则，要么移除 |
| **自主思考** | 这个功能是否能**从已有数据中发现模式、矛盾、缺口**？还是只是被动响应？ | 如果只是被动响应，考虑是否需要，或加推理层 |
| **自主进化** | 这个功能产出是否能够**修正系统自身行为**？是否有反馈环？ | 如果只是单向输出，考虑加回写环路 |

### 审计方法
1. 列出每个组件的**输入、处理、输出、反馈环**
2. 对每个组件回答三个自主问题
3. 标记不满足的组件：**裁剪？加固？加回写？**
4. 不裁剪的组件必须说明"为什么它的存在不需要自证价值"

## 设计原则

1. **容量满不是问题，是设计缺陷** — 如果 memory 或 user 满了，说明进化路径堵了或去重没做。不要加 cron 清理，要修写入路径
2. **cron 不是 janitor** — crons 做的是学习和反思，不是扫地
3. **进化路径比存储空间重要** — instinct 必须有毕业通道，否则 68→93% 的问题会反复出现
4. **user 和 peer card 是主从关系** — peer card 是事实来源，user profile 是它的本地缓存。写入时两者同步
5. **组件的存在价值必须被持续验证** — 没有使用率数据的组件应该被修剪。功能驱动而非需求驱动是过度设计的根因

## Pitfalls

1. **不要为 memory 膨胀建外部 cron 压缩** — 这是 Janitor 模式。正确做法：写入前自检 >80% 先合并，不需要外部 cron
2. **不要跳过 instinct 毕业** — ≥3 同域或 confidence≥0.85 自动进化。等待用户确认会导致堆积
3. **不要重复写入 user** — 同类画像走 replace 而非 add。去重是写入路径的一部分
4. **TMT 聚类内容质量** — 原始记忆是对话摘要，信念合成时需要去除 [user]/[assistant] 前缀
5. **CPU 嵌入推理慢** — llama-server 每次推理 12-15 秒，脚本 curl 超时设为 ≥45 秒，cron timeout ≥120 秒
6. **agent-mode cron 需要 memory tool** — 如果 cron 需要读写 memory（如反思报告），必须给 `enabled_toolsets: ["memory"]`
7. **没有使用率监控 = 死代码永远不被发现** — 每个 API 端点必须记录调用次数。没有数据就无法判断组件是否在发挥价值。这是本次架构反思最重要的教训
8. **功能驱动 ≠ 需求驱动** — "这个功能很酷"不等于"这个功能解决了一个实际需求"。AGE 图数据库和三馆闭环是最好的反面案例——设计和实现都正确，但需求不存在
9. **静默降级比崩溃更危险** — Deriver 的 tenacity 重试 3 次后静默丢弃失败任务。用户 API 正常、对话正常，但提炼管线已瘫痪。所有容错路径必须有显式告警
10. **组件的存在必须以"三个自主"为底线** — 不满足自主管理、自主思考、自主进化中至少一个的组件，应该被裁剪或加回写环路
