### D-205 外部校准源充分性建模

| 字段 | 内容 |
|:----|:-----|
| **来源** | 第九轮评析「强化项」（最高战略优先级）；认知基础§2.1他律性悖论的坦诚声明 |
| **预期版本** | v1.0 完成设计，v1.1 实现 |
| **改动量估值** | 大。需要建模：校准源的可靠性评估（多源一致性/历史准确率）、充分性判据（是否覆盖了需要校准的所有维度）、失败模式（单源失效/多源偏移/全源静默）、冗余策略。这是认知基础最大开放问题 |
| **不做的后果** | 体系的认知天花板受外部校准源供给边界钳制无法突破——「自演化」承诺是理论性的而非可操作的 |
| **当前阻碍** | 这是认知基础层面的开放问题，不是纯架构问题。需要认知层先给出充分性建模的方案 |

### D-206 四轴完整度量函数

| 字段 | 内容 |
|:----|:-----|
| **来源** | 第十轮评析 #1（认知完整性轴度量函数）；第九轮评析 III（多轴一致性） |
| **预期版本** | v1.1 |
| **改动量估值** | 中。在元认知层检测器族中增加认知完整性轴的三个度量指标监测器：(1) 反例覆盖度（已覆盖的推理空间边界/总边界）；(2) 路径禁区标注密度（已标注的死胡同路径/总探索路径）；(3) 组合约束网络连通性（约束间可达路径数/最大可能数） |
| **不做的后果** | 认知完整性轴仅有保护（is_structure）无度量，第四轴的价值独立性不可验证 |
| **当前阻碍** | 度量指标的定义需要在认知层确认计算方式 |

### D-207 元记忆偏置正反馈闭合机制

| 字段 | 内容 |
|:----|:-----|
| **来源** | 第十四份独立评析（2026-07-20，认知层面§1.2）；第九轮评析确认的 D.5 正反馈循环 |
| **预期版本** | v1.1 |
| **改动量估值** | 中。需要设计两个独立冗余的盲区测绘通道或一组对抗探针，在元记忆偏置监测器之外独立产生盲区地图，两条路径的差异信号作为偏置监测的输入。设计认知层应先确定盲区测绘的冗余策略 |
| **不做的后果** | 元记忆测绘偏置在两次外部校准之间无法自闭合——有偏的元记忆产出有偏的盲区地图，有偏地图触发有偏的重估，重估结果又强化元记忆偏置。系统在外部校准静默期间无法感知自身盲区正在扩大 |
| **当前阻碍** | 认知层仅承认此正反馈风险，未给出闭合机制的设计方向。需要认知层先确定冗余测绘通道的基本原理 |

### D-208 社会性校准信任模型认知层框架

| 字段 | 内容 |
|:----|:-----|

| **预期版本** | v2.0（前置条件：多 Agent 扩展需求明确） |
| **改动量估值** | 大。需要将社会性校准信任从「风险标注」推进到「认知层应对框架的雏形」：（1）多源校准的信任建模——不同校准源的可信度如何评估、衰减、重置；（2）见证价值轴在 multi-Agent 场景下的校准维度定义；（3）kairos://_social/ 路径空间的认知层语义定义 |
| **不做的后果** | 当多 Agent 扩展成为现实需求时，认知层对社会性校准的准备度不足——仅以空路径占位 + 三项激活前提应对，缺少认知框架层面的应对 |
| **当前阻碍** | 多 Agent 扩展在 v1.0 范围外，且需要架构层 _social/ 路径空间的实装经验做输入。当前阶段以「风险标注 + 占位」主动等待需求成熟 |

#### 用户画像 Schema 可配置（v1.1 规划）

将 `user_profiles` 表的提取 schema 从固定字段扩展为可配置模式——开发者可定义 profile 的提取字段、类型和抽取规则：

```yaml
# 示例配置：ai-companion.yaml
profile_schema:
  basic_info:
    fields:
      name: { type: string, required: true, extract_from: "自我介绍" }
      age: { type: integer, extract_from: "年龄相关对话" }
      occupation: { type: string, extract_from: "工作/职业相关" }
  interests:
    fields:
      hobbies: { type: string[], extract_from: "兴趣爱好对话" }
      preferred_topics: { type: string[], extract_from: "常聊话题" }
  psychological:
    fields:
      goals: { type: string, extract_from: "用户目标表达" }
      personality_traits: { type: string, extract_from: "性格特征" }
```

| 功能 | 说明 |
|:-----|:------|
| Schema 定义 | YAML/JSON 定义提取字段、类型、抽取来源提示 |
| 动态模板 | 不同场景使用不同 schema（心理咨询 vs 购物助手 vs 编程导师） |
