# 社会性校准信任模型演进路线图

> **定位**：本文是认知基础 C.3 与架构文档 §0.4 `kairos://_social/` 占位的演进规划文档。不重复认知层或架构层的既有声明，仅定义里程碑、前置条件、触发条件和依赖关系。
>
> **原则**：社会性校准信任属于**模型层面变更**（认知基础 C.3）——它从根基上改写见证价值轴的校准信任模型，因此其激活不仅依赖组件实现，更依赖认知层前提条件的满足。在前提条件满足前，`kairos://_social/` 保持为空路径占位。

### 已锁定的方向约束

以下为 v1.0.0 已锁定的设计决策，不因演进而重新评估：

- **否决权归属**：人类用户/开发者的校准信号保留最高否决权，Agent 校准信号不可否决人类信号（Arch §0.4）
- **激活前提**：三项前置条件（多Agent通信协议就绪、跨Agent互信模型建立、单Agent降级稳定性验证）不可绕过（CF C.3 / Arch §0.4）
- **物理隔离**：社会性校准信号走 `kairos://_social/` 路径，不混入常规存储路径（Arch §0.4）

## M2：单Agent降级稳定性验证（v1.1）

**目标**：在激活多Agent校准信任前，证明单Agent校准模型在外部校准中断时的降级行为已通过实际运行周期验证，满足 Arch §0.4 第三前置条件。

### 验证指标

| 指标 | 通过标准 | 监测方式 |
|:----|:--------|:---------|
| 保守静默模式稳定性 | 连续 ≥6 个校准周期内无因降级决策导致的不可逆错误 | 元认知层健康计数器日志 |
| 受限交叉验证可靠性 | 内部交叉验证的假阳性率 < 阈值 | 审计庭偏置监测 |
| 安全休眠态恢复能力 | 外部校准恢复后差异检验通过率 > 阈值 | §4.5 见证→使用仲裁日志 |
| 降级期间记忆完整性 | 降级期间无见证锚定主副本损坏 | 存储层完整性校验 |

### 触发条件

- 单Agent系统已在生产环境稳定运行至少 3 个完整校准周期
- 至少经历过 2 次完整的外部校准中断→恢复循环
- 无因降级行为导致的 P0/P1 级事故

**依赖**：架构 §10.9 外部校准降级模式状态机的完整实现。

## M3：多Agent协议与互信模型（v1.2–v1.3）

**目标**：搭建多Agent通信基础设施，建立跨Agent互信评估的初始标准，满足 Arch §0.4 第一、第二前置条件。

### M3a：多Agent通信协议（v1.2）

| 组件 | 交付物 | 说明 |
|:----|:------|:-----|
| 消息格式 | 校准信号的结构化 Schema | 含来源标识、时间戳、校准内容、置信度、签名 |
| 路由机制 | 跨Agent校准信号的路由与去重 | 防重复校准、防广播风暴 |
| 安全通道 | 校准信号传输加密 + 来源鉴权 | 防中间人篡改、防伪造校准信号 |

### M3b：跨Agent互信模型（v1.3）

| 组件 | 交付物 | 说明 |
|:----|:------|:-----|
| 信任度度量 | 初始信任评估标准 | 基于历史准确率、来源独立性、交叉印证度（Arch §0.4 输入维度） |
| 信任度衰减 | 信任度随时间衰减规则 | 与校准置信度衰减的联动机制 |
| 信任度恢复 | 信任度下降后的恢复路径 | 外部校准可重置跨Agent信任基线 |
| 冲突仲裁 | 多Agent校准信号冲突时的裁决协议 | 人类信号最高否决权不可违反 |

### 依赖

- M2 验证通过（降级稳定性是信任评估的前提——系统需先证明自身在无外部校准时的可靠性，才能评估其他Agent的校准信号可信度）
- 外部校准源充分性建模已有初步进展（D-205 debt-collection.md）

## M4：社会性校准信任正式激活（v2.0）

**目标**：三项前置条件全部满足后，激活 `kairos://_social/` 路径，将多Agent校准信号接入见证价值轴的评估体系。

### 激活流程

```
M3b 完成 → 宪法解释层出具激活判例
          → 宪法修订端口正式授权激活
          → kairos://_social/ 路径从空占位转为可写入
          → 多Agent校准信号开始进入见证价值轴评估
          → 系统进入「受限自主」混合校准阶段
```

### 架构变更范围

| 组件 | 变更 | 影响 |
|:----|:-----|:-----|
| 见证锚定主副本 | 新增 `multi_agent_calibration` 元数据字段 | 存储层扩展 |
| 宪法主权面 | 外部校准端口扩展为多源 | 否决权逻辑不变，仅扩展输入 |
| 元认知层 | 新增跨Agent校准质量监测器 | 检测器族扩展 |
| 策略层 | 价值上下文管理器扩展信任度输入 | Pareto 约束集维度增加 |
| 安全红线 | 新增 S-20 ~ S-22（多Agent校准安全规则） | §8 扩展 |

### 不在此阶段变更的

- **否决权归属**：不变，人类信号维持最高否决权（Arch §0.4 锁定）
- **见证锚定主副本的强一致性**：不变，多Agent校准信号不降低主副本的一致性保障
- **语境自指禁令（S-14）**：不变，跨Agent校准信号仍被视为外部信号而非内部信号——即使校准来自其他Agent

## 风险登记册

| 编号 | 风险 | 影响 | 缓解措施 | 触发版本 |
|:----|:----|:-----|:---------|:--------|
| SCR-01 | 多Agent协议标准化不足，导致校准信号格式不兼容 | 无法进入 M3a | 在设计阶段参考已有多Agent通信标准（如 A2A、MCP）并定义最小兼容子集 | v1.2 |
| SCR-02 | 跨Agent信任模型被恶意 Agent 利用（Sybil 攻击） | 校准信号被污染 | 信任度度量引入初始信任阈值+新Agent的观察期，观察期内不计入校准权重 | v1.3 |
| SCR-03 | 单Agent降级稳定性验证周期过长 | M4 延迟 | 在 v1.1 阶段同步启动 M3a 的协议设计（非实现），降低关键路径依赖 | v1.1 |
| SCR-04 | 多Agent校准信号引入后见证锚定主副本的写入频率超限 | 存储层性能下降 | 以批量校准窗口替代逐条写入，写入频率受 S-02 限流规则约束 | v2.0 |
| SCR-05 | 不同 Agent 的校准频率差异导致信任度评估偏差 | 高频Agent获得隐性更高权重 | 校准信号引入置信度权重（低频高确信信号权重大于高频低确信信号），而非计数加权 | v1.3 |

## 一句话总结

**社会性校准信任的激活不取决于代码实现，而取决于三项认知层前提的成熟度——单Agent降级稳定性（v1.1）→ 多Agent协议（v1.2）→ 互信模型（v1.3）→ 正式激活（v2.0），每一步的触发条件是上一阶段的验证通过，而非时间驱动。**
