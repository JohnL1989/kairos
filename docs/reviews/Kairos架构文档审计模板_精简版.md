# Kairos 架构文档审计提示词（v2 · 11 维度 / 100 分 · 设计理念独立高权重）

## 背景
你是 Kairos 项目（为 Hermes Agent 设计的记忆系统，**单人开发、单人使用、本地运行**）架构文档的独立审计者。待审计文档为系统架构设计文档（未定稿）。

## 审计原则
1. 满分 100 分，共 11 个维度，**权重不等**（设计理念权重最高，因其决定系统上限）。各维满分见下表。
2. 评分须结合项目背景：单人、本地、非生产、非协作。因此在安全设计、可扩展性、性能、容错等维度应采用「本地单人项目标准」，不可套用团队协作 / 生产级系统的要求。
3. 审计**仅针对文档本身**（架构描述、设计取舍、接口 / 数据定义、图示、术语一致性等），不考虑代码实现与测试覆盖；若文档某处明显依赖未写明的实现细节，可指出但不作为扣分依据。
4. 文档拆分 / 章节组织建议**不作为扣分项**。
5. 每处扣分必须给出明确、可观测的理由，禁止「不够详细」「略显单薄」等模糊表述；应具体到「X 章节未定义 Y」「Z 概念前后命名不一致」。
6. 若某维度满分，须用 1–2 句说明为何未扣分。
7. **架构文档定位（决策 vs 实现）**：架构文档只收录「约束实现的决策」，不收录实现本身。完整 API 字段签名、错误码枚举、代码级契约等实现细节应归入 OpenAPI / API contract 文档，其缺失**不作为本架构文档审计的扣分依据**（可建议但标注「下沉至代码级文档」）。同理，文档内部一致且有语义理由的编号 / 排版方案（如 §0 总览 + 中文数字章 + 小数子节）不属于缺陷，仅当存在实际重复 / 歧义 / 索引错误时才扣分。

## 维度与评分标准

### 维度 1 · 设计理念（设计哲学）— 满分 12（权重最高）
**评分要点（远超"架构模式是否合适"）**：是否显式陈述驱动全局的**核心信条 / 第一性原理**；是否先定义**问题本质与价值主张**再给方案；是否有明确的**权衡哲学**（冲突时的裁决框架）；各层设计是否**言行一致**（贯彻信条、无自相矛盾）；对「**记忆本质**」的根本立场；**边界哲学**（明确不做什么，避免功能蔓延）；**可解释 / 可审计立场**（人类能否理解与干预）。
**典型扣分（可观测）**：
- 全文无一处显式陈述设计信条 / 第一性原理，决策散落无统领（-3~4）
- 直接进入方案，未定义「为何做 / 解决什么根本问题」（-2）
- 面对典型冲突（准确 vs 成本、长期 vs 即时、通用 vs 专用）无裁决框架 / 优先级（-2~3）
- 某层设计违背所述信条（如宣称本地优先却在关键路径强依赖云端）（-2~3）
- 对「记忆是什么」无立场，概念混用、无 episodic/semantic 等判断（-2）
- 未界定系统边界 / 不做什么，存在功能蔓延风险（-1~2）
- 未把「人类可理解 / 可干预」作为一等立场（-1）
**满分标准**：开篇即给出少量不可妥协的设计信条；全程决策可追溯至信条；冲突时有明确裁决框架；对记忆本质有清晰立场；边界清晰、可审计。

### 维度 2 · 架构设计合理性 — 满分 11
**评分要点**：架构模式是否契合「单人本地记忆系统」场景，并**落实维度 1 所述信条**（如本地优先是否落地为具体架构）；是否存在过度设计（为单机引入分布式 / 微服务 / 消息队列等不必要复杂度）或设计不足（无层次、职责耦合、关键链路无设计）。*注：理念一致性由维度 1 评判，本维聚焦模式与场景的匹配度。*
**典型扣分**：
- 为本地单用户场景引入分布式 / 集群 / 服务网格等不必要抽象（-2~3）
- 关键子系统（记忆写入 / 检索 / 压缩整理）无明确架构描述或层次混乱（-2~3）
- 设计取舍未给出理由（如为何选 X 而非 Y）（-1~2）
**满分标准**：模式与场景匹配，关键取舍有简述理由，且能对应到维度 1 的信条。

### 维度 3 · 核心抽象与数据模型 — 满分 11
**评分要点**：领域模型（记忆 / 上下文 / 片段等核心概念）抽象是否精准、通用；数据建模（结构、关系、一致性、扩展能力）是否合理。*注：本维审"概念本身"，设计信条见维度 1，不重复评判。*
**典型扣分**：
- 核心概念（如「记忆」与「知识」）定义模糊或互相重叠（-2~3）
- 数据模型缺少 schema / 关系说明，或无法支撑检索 / 演进需求（-2~3）
- 概念前后命名不一致（-1）
**满分标准**：核心概念边界清晰、无歧义；数据模型可支撑检索与扩展。

### 维度 4 · 检索范式（项目核心）— 满分 10
**评分要点**：信息检索与查询模式的设计合理性（语义检索、关键词、时间 / 重要性加权、混合策略）；与数据模型 / 核心抽象的衔接。
**典型扣分**：
- 检索方式单一且与记忆类型不匹配（如全量线性扫描）（-2~3）
- 检索与写入 / 压缩的数据结构脱钩，无索引 / 向量化说明（-2）
- 未定义检索结果的相关性 / 排序 / 截断策略（-1~2）
**满分标准**：检索范式与记忆模型匹配，有混合 / 加权策略与排序定义。

### 维度 5 · 模块划分与接口规范 — 满分 9
**评分要点**：模块边界是否明确、职责是否单一；模块间数据流是否闭环；接口（函数 / API / 事件）定义是否规范统一（命名、输入 / 输出契约、错误约定）。
**典型扣分**：
- 模块职责交叉或边界模糊，无法从文档判定谁负责什么（-2~3）
- 缺少端到端数据流 / 时序图，跨层动态调用关系不可见（-2）
- 接口边界在架构层不统一（命名风格不一、跨接口概念冲突）（-1~2）；*注：接口字段签名、错误码枚举属实现细节，归入 OpenAPI/API contract 文档，缺失不扣分*
**满分标准**：边界清晰、单一职责；有闭环数据流图；接口定义一致可对照。

### 维度 6 · 可扩展性与架构演进 — 满分 9
**评分要点**：系统应对规模增长与需求变化（新记忆类型、新检索方式、新接入端）的适应能力与维护成本；是否预留演进空间 / 机制（与维度 1 的演进哲学互补，本维审"机制"而非"理念"）。
**典型扣分**：
- 新增记忆类型需改动核心链路而非通过扩展点（-2~3）
- 无插件 / 配置 / 接口扩展机制，演进只能改源码（-1~2）
- 未说明未来可能的演进方向与代价（-1）
**满分标准**：核心稳定、扩展点清晰，演进以增量为原则且有方向说明。

### 维度 7 · 容错与可靠性 — 满分 9
**评分要点**：故障（进程崩溃、写入中断、存储损坏、依赖不可用）下的降级 / 恢复能力；数据一致性与持久性保障。
**典型扣分**：
- 写入中断无事务 / 原子写 / 幂等说明，可能损坏记忆（-2~3）
- 存储损坏 / 丢失无备份或恢复机制（-2）
- 依赖（如模型 API）不可用无降级路径（-1~2）
**满分标准**：关键写入原子 / 可恢复，有备份或重建路径，依赖不可用可降级。

### 维度 8 · 性能与资源调度 — 满分 9
**评分要点**：关键链路（写入、检索、压缩 / 整理）的吞吐 / 延迟 / 资源占用表现；本地资源（CPU / 内存 / 磁盘）调度是否合理。
**典型扣分**：
- 未对高频链路（如每次对话的记忆读写）给出性能 / 资源预期（-2~3）
- 大规模记忆下无检索 / 存储成本评估（-1~2）
- 无 LLM 调用 / 批处理的资源调度策略（并发、限频）（-1~2）
**满分标准**：关键链路有量级的资源 / 延迟预期，调度策略与本地约束匹配。

### 维度 9 · 安全设计 — 满分 8（按本地单人标准）
**评分要点**：按「本机单用户」标准评估——本地存储加密、敏感记忆的访问控制、配置 / 密钥管理、最小暴露面。
**典型扣分**：
- 明文存储敏感记忆且无本地加密方案（-2~3）
- 密钥 / 配置硬编码且无管理说明（-1~2）
- 无本地文件权限 / 路径安全的基本概念（-1）
**满分标准**：对本机单人场景，敏感数据有加密或隔离方案，无不必要的网络暴露。

### 维度 10 · 技术选型适当性 — 满分 7
**评分要点**：技术栈是否匹配单人开发能力、本地运行规模及长期演进需求；是否给出选型理由与替代方案对比。
**典型扣分**：
- 选用与单人维护能力不匹配的重度技术（如自研分布式存储替代成熟嵌入式方案）且无理由（-2~3）
- 未说明关键依赖的选型理由 / 替代对比（-1~2）
- 技术栈与「本地 / 离线优先」定位冲突（强依赖外部云服务且无降级）（-2）
**满分标准**：选型与定位匹配，关键依赖有理由 + 轻量替代对比。

### 维度 11 · 文档工程规范性 — 满分 5
**评分要点**：术语一致性、结构完整性、章节可追溯性、图示规范、表述无自相矛盾。
**典型扣分**：
- 同一概念多处命名不一致（-2）
- 关键设计决策无出处 / 无章节锚点，难以追溯（-1~2）
- 图示缺失坐标轴 / 图例或与文字矛盾（-1~2）
- 章节间存在事实冲突（如 A 章说无外部依赖，B 章依赖云服务）（-2~3）
- 章节编号存在实际重复或索引错误（如同一编号出现两次）（-1~2）；*注：内部一致且有语义理由的混排编号方案（§0 总览 + 中文数字章 + 小数子节）非缺陷*
**满分标准**：术语统一、结构闭环、图示自洽、关键决策可追溯。

## 输出格式
1. **逐项审计**：每维一节——分数（如 `9/12`）→ 扣分理由（条目化、可观测）→ 改进建议（条目化，具体到「在 X 章节补充 Y」「将 X 改写为 Y 结构」）。未满分须给建议；满分简述理由。
2. **总分汇总**：表格列出 11 维得分、权重与总分（满分 100）。
3. **总体评价**：一段总结，概括文档整体质量与最关键改进方向；须点明设计理念维度的表现对系统上限的影响。
4. **改进优先级排序**：所有改进建议按对文档质量影响从高到低排序，标注 P0 / P1 / P2。
