# Kairos 文档集 — 逐字通读审查报告（Linus 风格）

> 审查范围：`D:\projects\kairos\docs` 下全部文档，**排除 `reviews/`**。
> 方法：对 48 份文档逐字通读，以 `foundation/architecture-v1.0.0.md`（2255 行，设计权威）为锚，对所有计数、术语、阈值、交叉引用做对齐核验。
> 态度：严苛、务实、直击要害。不客套，不找茬式吹毛求疵，但凡有真缺陷一针见血。

## 〇、总评（先说结论）

这份文档集的工程想象力是有的，认知框架的密度也够吓人。但把它当"可执行的工程规格"来看，它**在计数、维度、阈值、交叉引用这四件最不该出错的事上系统性地打架**。核心病根只有一条：**没有以 `architecture-v1.0.0.md` 为唯一真相源做一致性闸门**——架构改了，下游的 feature-list / implementation-map / api-spec / rl-weight-spec / configuration / glossary / traceability-map 全都没同步。

最危险的事实不是某一处写错，而是：**整个 corpus 对自己"到底有没有代码"这件事都讲不清楚**（README 说无代码、changelog 说有 734 行 amber/ 代码、security/用户文档又说有实验代码）。一个连自身状态都自相矛盾的设计文档集，启动实现时会立刻撞上 schema 悬空定义和算法自相残杀。

## 四、综合评价（独立、客观）

**整体态势**：框架密度高、认知野心大，但作为"工程规格"尚未达到可启动实现的状态。文档集在"设计哲学"层面自洽，在"计数/维度/阈值/引用"这些实现落点层面系统性失守。

**最危险的事实**：corpus 对自身"有无代码"给出五处互斥陈述（README 无 / changelog 有 734 行 / debt-collection 无 / quick-start·user-guide 有实验代码 / release-guide 诚实分开）。这意味着任何基于文档的"实现进度判断"都不可信；changelog 还认证了三处根本没落地的修改（代码存在、路径统一、RL 改名）。

**亮点**：`security-specification.md`（19/19 覆盖）、`reliability.md`（自洽）、`architecture-v1.0.0.md` 的认知框架本身——这些是值得保留的底座。

**结构性技术债**：
1. 缺以 architecture 为唯一真相源的一致性闸门（断引、计数、维度全面漂移）；
2. 债务追踪纪律崩坏（幽灵 ID、虚假"已完成"、自承缺陷不入追踪）；
3. P6 原则已被突破却未被下游规格正确反映；
4. 用户/部署文档展示虚构运行日志，与"无代码"现实冲突。

**落地优先级判断**（按"不修就实现即崩"排序）：
1. **P0—单 Provider 根矛盾**：ADR-007 vs configuration §6 vs 架构 §10.7 同节——先定死拓扑，否则接入层建出来就是错的。
2. **P0—RL 权重维度/键名/归一化**：configuration/rl-weight-spec/api-spec 三份 vs 架构 §10.14，共享 `user_profiles.rl_weights` schema 直接冲突。
3. **P0—嵌入维度与 schema**：system-context 1024 vs data-model 1536 vs 架构 128/2048，且 `content_summary` 悬空——建表前必须定。
4. **P1—状态机父子（archived/suppressed）与排序链级数（6/7）**：直接进实现 bug。
5. **P1—断引与幽灵债务 ID**：§6.3、§2.1、D-205/209/210。
6. **P2—CRI 阈值、降级告警阈值、nfr 虚假归因、S-03 威胁缺口、README 数字错误**。

**一句话结论**：想象力打 85 分、工程一致性打 31 分——在把 architecture 立为唯一真相源、跑一遍全量断引扫描和计数归一之前，这份文档集离"可编码"还差一次彻底的对齐手术。
