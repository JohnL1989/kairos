# Kairos 使用场景

> **定位**：描述 Agent 与 Kairos 交互的典型场景。功能清单说「能做什么」，用例说「在什么情境下按什么顺序做」。

---

## 场景 1：冷启动——首次运行与种子注入

**场景**：用户首次启动 Kairos，系统处于冷启动阶段。

**流程**：
1. 用户通过 CLI 执行 `kairos init`（或首次运行时自动触发）
2. 宪法主权面的外部校准端口读取环境变量中的种子锚点配置
3. 种子锚点注入存储层（标记 `provenance=system_generated`）
4. 元认知层记录种子生命周期初始状态（最小化检查通过）
5. 系统进入冷启动期——探索预算初始分配为最大值

**验证**：写入一条记忆 → 检索成功 → 路径空间可访问

## 场景 2：日常写入与检索

**场景**：Agent 在运行过程中写入新经验并检索已有记忆。

**流程**：
1. Agent 通过 Tool `memories_write` 向路径 `kairos://sessions/{id}/` 写入记忆
2. 接入层验证输入（长度、路径格式、API Key）
3. 存储层创建记忆记录 + 提交嵌入计算（异步巩固管道执行，嵌入就绪前检索降级为路径前缀匹配）+ 写入路径索引
4. 使用事件总线发送 `use_event` 事件
5. 后续检索：Agent 调用 `memories_search` 按路径前缀或语义查询
6. 结果经权重调制后返回

**变量**：
- 指定契约类型（permanent/ondemand/environmental/temporary）
- 附带情感 VAD 元数据
- 标注与其他记忆的关系

## 场景 3：升华触发与审批

**场景**：系统空闲期间自动执行升华，将原始经验提纯为行为规则。

**流程**：
1. 调度器检测到系统空闲（无活跃检索/写入持续 N 周期）
2. 从使用负载低的记忆中选取升华候选（`provenance!=exploration`）
3. raw → item：提取核心模式，去语境化
4. item → strategy：归纳为可复用的策略
5. strategy → behavior：生成行为规则，进入 `awaiting_approval` 状态
6. 用户收到审批通知 → 手动确认或拒绝
7. 确认后 behavior 生效，标记 `provenance=internal_inference`
8. 升华日志写入使用事件总线

## 场景 4：外部校准介入

**场景**：用户/外部系统发现记忆的见证锚定不准确，发送校准信号修正。

**流程**：
1. 用户通过 REST API `POST /v1/calibrate` 发送校准信号（memory_id + score + source）
2. 宪法主权面外部校准端口接收并鉴权
3. 校准信号写入见证锚定主副本（`narrative_coherence_score` 更新）
4. 差异检验：使用权重影子副本与更新后的见证锚定比对
5. 若无冲突：校准信号注入完成
6. 若有冲突：使用权重标记 `suspect_flag=true`，转入沙箱验证环
7. 事件总线广播 `calibration_signal` 事件

## 场景 5：遗忘与复兴

**场景**：系统自动遗忘低价值记忆，被遗忘的记忆因再次需要而复兴。

**流程**：
1. 遗忘调度器扫描存储层：计算每条记忆的遗忘得分
2. 得分 > 阈值的记忆进入遗忘队列
3. 遗忘操作执行（标记 `status=suppressed`——检索抑制态，非删除）——不同于 `archived`（冷存储态），suppressed 抑制但可复兴，archived 压缩但不抑制检索
4. 一段时间后：Agent 检索了一条已归档的记忆（通过路径或语义）
5. 遗忘调度器检测到该记忆再次被检索 → 触发复兴加速通道
6. 记忆恢复为 `status=active`，激活权重设为初始值

## 场景 6：外部校准中断与降级

**场景**：长时间无外部校准信号到达，系统进入自动降级。

**流程**：
1. 元认知层健康计数器监测外部校准端口的信号间隔
2. 中断持续 < N 周期：**保守静默**——使用权重停止积累，仅处理显式检索
3. 中断持续 N-M 周期：**受限交叉验证**——允许高置信度记忆间关联验证
4. 中断持续 > M 周期：**安全休眠**——暂停主动推理，仅维持数据完整性
5. 外部校准信号恢复 → 所有 `provisional_during_calibration_gap` 标记产物进入差异检验

（N、M 为配置参数，详见 `architecture-v1.0.0.md §10.9` 降级状态机与 `ops/configuration.md` 中 `DEGRADATION_PERIOD_N`/`DEGRADATION_PERIOD_M`）
6. 未通过者回滚 → 通过者正常合并 → 系统退出降级模式

## 场景 7：安全红线触发

**场景**：S-14 触发——使用权重试图反向写回见证锚定。

**流程**：
1. WM 层处理检索结果时，发现某记忆的使用权重陡升
2. 差异检验流程启动，但合并逻辑错误地尝试写回见证锚定
3. 存储层检测到写回操作 → 触发 S-14 安全红线
4. 操作被拒绝，审计日志记录时间戳/操作者/目标记忆/红线编号
5. HMAC 链更新
6. 元认知层偏置监测器收到告警（来源收敛 + 校准衰减检查）
7. 宪法解释层评估是否需要人工介入

## 场景 8：种子退化

**场景**：种子锚点在系统运行时因探索-种子冲突而触发退化。

**流程**：
1. 探索活动产出的记忆与种子锚点的价值判断持续冲突
2. 冲突信号累积 → 达到预设阈值
3. 元认知层种子生命周期追踪触发权重衰减评估请求
4. 宪法解释层裁定：执行提前退化
5. 种子锚点权重从特殊值衰减为普通置信度级别
6. 偏置重置选项提供（清除种子衍生的累积偏置，保留审计基线）
7. 日志写入使用事件总线，标记 `exploration_budget`
